## PHASE 3 — SECTIONS VS COMPONENTS — THE DECISION RULE

Apply this rule strictly for every UI piece you encounter:

GOES IN sections/[page]/       if ALL of these are true:
  - It represents a full visual band or block of a specific page
    (e.g. Hero, Features, Pricing, Stats, ActivityFeed, CTA banner)
  - It is only used on ONE specific page
  - It is too large or too page-specific to be a generic component
  - It may fetch its own data or contain business logic for that page area

GOES IN components/common/     if ALL of these are true:
  - It has no business logic tied to a specific page
  - It is (or could be) reused across 2+ pages or sections
  - It accepts props rather than pulling from a page-specific context
  - Examples: Button, Input, Modal, Tooltip, Badge, Spinner, Card, Avatar,
    Dropdown, Table, Tabs, Alert, SectionWrapper, Heading

GOES IN components/layout/     if:
  - It wraps pages or sections structurally
    (Navbar, Sidebar, Footer, PageWrapper, AuthLayout, DashboardLayout)

GOES IN components/[feature]/  if:
  - It is reusable within one feature/domain but not page-level
    (e.g. UserCard used across user-related sections/pages)

A page in pages/ should be a THIN SHELL — it imports and composes sections:

  // pages/Home.jsx
  import HeroSection from '@sections/home/HeroSection';
  import FeaturesSection from '@sections/home/FeaturesSection';
  import TestimonialsSection from '@sections/home/TestimonialsSection';

  export default function Home() {
    return (
      <>
        <HeroSection />
        <FeaturesSection />
        <TestimonialsSection />
      </>
    );
  }

If a section is used on more than one page, promote it to components/[feature]/ or components/common/.

## PHASE 5 — RULES FOR EVERY FILE MOVE

When moving or creating any file, you MUST:

1. Update every import path that references it across the entire codebase.
2. Verify the new import path resolves correctly (use @/ aliases if supported).
3. Ensure the moved component/section still renders correctly — props, context, hooks all intact.
4. Register any new page in src/routes/index.jsx immediately.
5. Never leave a dangling import or a broken reference.
6. Never use placeholder JSX (no empty returns, no // TODO handlers).
7. Consolidate any duplicate logic into a single source.

## PHASE 6 — FIX RENDERING & ARCHITECTURE ISSUES

After reorganizing:
- Every page must be reachable via the router (check routes/index.jsx).
- Every page must compose its sections in the correct render order.
- Every section must receive all required props — trace the chain: route → page → section → component.
- Every API call must be in services/, called from a hook in hooks/, consumed by a section or page.
- Side effects belong in hooks, not in component or section bodies.
- Shared state belongs in context/ or store/, not drilled as props 3+ levels deep.
- Remove any inline fetch() or axios calls inside component or section files.
- Remove any hardcoded URLs — use environment variables (import.meta.env.VITE_* for Vite).
